Introduction

Accommodation platform operators and distribution platform operators must file GST/HST information returns with the Canada Revenue Agency (CRA), as outlined by sections 211.21 and 211.25 of the Excise Tax Act (ETA).

The DPIR schema is based on schema introduced by the Organisation of Economic Co-operation and Development (OECD) Model Rules for Reporting by Platform Operators (“Model Rules”). The DPIR schema includes the applicable Model Rules data elements and introduces new data elements only when needed.

The DPIR schema user guide is similar to the Model Rules user guide, both in structure and content. This makes filing information returns with the CRA similar to reporting to other tax administrations using the OECD's Model Rules.

Overall document structure

This user guide is organized around the main components of the XML document. We refer to the XML document as the "DPIR message" or "message" for short.

Figure 1 shows a high-level overview of a DPIR message. It includes these data elements:

  • Message Specification (MessageSpec)
  • Platform Operator (PlatformOperator)
  • Reportable Seller (ReportableSeller)
  • Identity
  • Relevant Activities (RelevantActivities)
  • Document Specification (DocSpec)

The user guide refers to XML elements using their full name (for example, "Message Specification") but provides its XML name in brackets (MessageSpec) for clarity.

Figure 1: Overall document structure

The root of the DPIR message is a DPIR data element. Each DPIR message must contain a Message Specification and a DPIR Body (DPIRBody). It may optionally contain a version attribute, which specifies the version of the schema to which the message complies.

A DPIR Body includes data about the Platform Operator and its Reportable Sellers. Each Reportable Seller includes:

  • the Identity data element, which describes who the Seller is
  • Relevant Activities (RelevantActivities), which describes services provided by the Seller, earnings (or consideration) derived from those services and related taxes

The DPIR schema also provides a mechanism to uniquely identify Platform Operator and Reportable Seller data elements.

More details are in the Identification of data elements.

A DPIR message must have exactly one DPIR Body data element, even though the DPIR schema allows for any number of DPIR Body data elements or none at all. The more general message structure in Figure 1 aligns the DPIR schema with the Model Rules schema, and provides for a wide range of applications for consistency with international standards.

Schema elements

Message Specification type

The Message Specification (MessageSpec) type represents a message header. It includes the Platform Operator identifier (SendingEntityIN), the time when the message was created, the Reportable Period, the nature of the data in the message (original, correction or deletion), contact information, and other data elements.

DurationClass attribute

Indicates if the rental is a short-term accommodation (defined as a rental of less than one month) or long-term accommodation (defined as a rental of more than one month). It can have one of the following values:

  • DPIR101 = Short-term accommodation (continuous rental occupancy of less than one month)
  • DPIR102 = Long-term accommodation (continuous rental occupancy of at least one month)
  • DPIR103 = Short-term and long-term accommodation combined
  • DPIR104 = Rental occupancy of unknown duration

If the Duration Class is not specified, then the value of DPIR101 is assumed.

Size: enum

Input Type: dpir:RentalDurationClass_EnumType

Requirement: Optional

Note: A duration of one month is defined as a duration that starts on a particular date in one month and ends right before the same date in the following month. If the following month does not have a corresponding date, then the last date in the month is considered.

Some examples of duration of one month are: Jan 15 – Feb 14, Jan 31 – Feb 28 for a non-leap year, Jan 31 – Feb 29 for a leap year. In these examples, the start and the end dates are included in duration.

AnnualConsideration data element

Identification of data elements

In each DPIR message, the Platform Operator and Reportable Seller data elements must have unique identifiers. An identifier of a DPIR message is specified in the Message Reference ID (MessageRefId), while the Platform Operator's and Reportable Seller's identifiers are specified in the Document Reference ID (DocRefId), which is found in the Document Specification (DocSpec) data element that each Platform Operator and Reportable Seller contain.

In this section, we first describe the format of Message Reference ID and Document Reference ID. Then, we detail the Document Specification (DocSpec) data element, and outline how it applies to Platform Operators and Reportable Sellers.

Format of Message Reference ID

A DPIR message identifier is specified in the Message Reference ID (MessageRefId) data element, and has the following format:

<transmitting country><reporting year><receiving country>DPIR<BN #>M<message root>

The meaning of the Message Reference ID components is as follows:

  • <transmitting country> is a 2-character ISO code of the country where the Platform Operator is located
  • <reporting year> is the year of the Reporting Period, as specified in the Message Specification data element
  • <receiving country> is a 2-character ISO code of Canada. This code must be "CA"
  • DPIR is a fixed literal
  • <BN #> is a BN9 of the Platform Operator
  • M is a fixed literal
  • <message root> is an 8-digit number that completes the Message Reference ID and ensures its uniqueness in time and space. It could be a randomly generated number or incremented from the message root of the previous Message Reference ID

For example, US2023CADPIR123456789M11110201 is a valid Message Reference ID. It means that:

  • The Platform Operator is located in the USA ("US"), is filing the information return to Canada ("CA") for the calendar year 2023 ("2023")
  • Information return is filed as part of the Digital Platform Information Returns ("DPIR") program
  • The BN9 of the Platform Operator is "123456789"
  • Message root is "11110201"

Format of Document Reference ID

Identifiers of Platform Operators and Reportable Sellers are specified in the Document Reference ID (DocRefId) data element, which is found in the Document Specification (DocSpec) data element. Data elements containing the DocSpec data element are called correctable data elements.

The following is the format of the Document Reference ID:

<Message Reference ID><data element identifier>

<Message Reference ID> represents a DPIR message ID and is detailed in the previous section. The <data element identifier> is a six-digit number generated by the Platform Operator that, along with <Message Reference ID>, uniquely identifies each data element in space and time.

The Document Specification data type defines the structure for identification of data elements, and it specifies the type of corrections that can be made on those data elements.

Document Specification data element for Platform Operators

The following rules govern the use of the Document Specification data element for Platform Operators:

  • Platform Operator data elements cannot be deleted. Therefore, the only acceptable values for the Document Type Indicator (DocTypeIndic) are OECD0, OECD1 and OECD2
  • The Document Type Indicator will be set to OECD0 when a DPIR message contains updates of seller data (amendments, corrections or deletions) but the Platform Operator data element does not change
  • If a Platform Operator data element was filed with incorrect data, then a new information return must be filed with correct Platform Operator data. The new Platform Operator data element will have the Document Type Indicator set to OECD2 and will replace the Platform Operator identified in the CorrDocRefId data element
  • If a DPIR message exceeds the 150MB size limit and has to be split into several smaller messages, then each Platform Operator data element in the smaller messages will have unique Document Reference ID

Document Specification data element for Reportable Sellers

The following rules govern the use of the Document Specification data element for Reportable Sellers:

  • Seller's data can be corrected (OECD2) or deleted (OECD3)
  • If the Document Type Indicator is set to OECD2, then the Reportable Seller data element replaces the Reportable Seller identified in the CorrDocRefId data element
  • If the Document Type Indicator is set to OECD3, then Relevant Activities within the Reportable Seller data element must not contain any data elements. This would indicate that the data for the Reportable Seller specified in CorrDocRefId should be deleted

Any past filed information returns can be modified, as long as the modifications being filed do not impact subsequent returns. If they do, a correction should be filed for every impacted return in reverse order. In other words, start with the most recently filed information return, and end with the oldest.
