Introduction

Accommodation platform operators and distribution platform operators must file GST/HST information returns with the Canada Revenue Agency (CRA), as outlined by sections 211.21 and 211.25 of the Excise Tax Act (ETA). These information returns are to be filed in XML, according to the Digital Platform Information Returns (DPIR) schema, as described here.

The DPIR schema is based on schema introduced by the Organisation of Economic Co-operation and Development (OECD) Model Rules for Reporting by Platform Operators (“Model Rules”). The DPIR schema includes the applicable Model Rules data elements and introduces new data elements only when needed.

The DPIR schema user guide is similar to the Model Rules user guide, both in structure and content. This makes filing information returns with the CRA similar to reporting to other tax administrations using the OECD’s Model Rules.

Overall document structure

This user guide is organized around the main components of the XML document. We refer to the XML document as the “DPIR message” or “message” for short.

Figure 1 shows a high-level overview of a DPIR message. It includes these data elements:

  • Message Specification (MessageSpec)
  • Platform Operator (PlatformOperator)
  • Reportable Seller (ReportableSeller)
  • Identity
  • Relevant Activities (RelevantActivities)
  • Document Specification (DocSpec)

The user guide refers to XML elements using their full name (for example, “Message Specification”) but provides its XML name in brackets (MessageSpec) for clarity.

Placeholder image: Figure 1: Overall document structure

The root of the DPIR message is a DPIR data element. Each DPIR message must contain a Message Specification and a DPIR Body (DPIRBody). It may optionally contain a version attribute, which specifies the version of the schema to which the message complies.

A DPIR Body includes data about the Platform Operator and its Reportable Sellers. Each Reportable Seller includes:

  • the Identity data element, which describes who the Seller is
  • Relevant Activities (RelevantActivities), which describes services provided by the Seller, earnings (or consideration) derived from those services and related taxes

The DPIR schema also provides a mechanism to uniquely identify Platform Operator and Reportable Seller data elements.

More details are in the Identification of data elements section.

A DPIR message must have exactly one DPIR Body data element, even though the DPIR schema allows for any number of DPIR Body data elements or none at all. The more general message structure in Figure 1 aligns the DPIR schema with the Model Rules schema, and provides for a wide range of applications for consistency with international standards.

Schema elements

Message Specification type

The Message Specification (MessageSpec) type represents a message header. It includes the Platform Operator identifier (SendingEntityIN), the time when the message was created, the Reportable Period, the nature of the data in the message (original, correction or deletion), contact information, and other data elements.

Image placeholder: Figure 2: Message Specification type

The following data elements are used to define the Message Specification type.

SendingEntityIN data element

Identifies the Platform Operator. This must be a BN15 issued to the Platform Operator by the Canada Revenue Agency (CRA).

Size:
1 to 200 characters
Input type:
stf:StringMin1Max200_Type
Requirement:
Mandatory

TransmittingCountry data element

Identifies the country from where the Platform Operator is sending the DPIR message. It uses a 2-character alphabetic country code based on the ISO 3166-1 Alpha 2 standard.

Size:
2-character
Input type:
iso:CountryCode_Type
Requirement:
Mandatory

ReceivingCountry data element

Identifies the country receiving the DPIR message. It is used to facilitate further potential alignment of the information returns with the Model Rules. The value of this data element must be “CA”.

Size:
2-character
Input type:
iso:CountryCode_Type
Requirement:
Mandatory

MessageType data element

Specifies the type of the DPIR message. The value of this data element must be “DPIR”.

Size:
enum
Input type:
dpir:MessageType_EnumType
Requirement:
Mandatory

Warning data element

Free text field, which allows input of specific cautionary instructions about the use of the DPIR message content.

Size:
1 to 4,000 characters
Input type:
stf:StringMin1Max4000_Type
Requirement:
Optional

Contact data element

Free text field allowing input of specific contact information relating to the Platform Operator. Contact data element should include an email address.

Size:
1 to 4,000 characters
Input type:
stf:StringMin1Max4000_Type
Requirement:
Mandatory

MessageRefID data element

Uniquely identifies an XML document or DPIR message containing the information returns that are filed by the Platform Operator. This identifier allows both the sender and receiver to identify the specific message at a subsequent point in time, if questions arise. The format of the Message Reference ID is specified in the Identification of data elements section.

Size:
1 to 170 characters
Input type:
stf:StringMin1Max170_Type
Requirement:
Mandatory

MessageTypeIndic data element

Specifies the type of information being sent. This can either be a new information return filed for the first time for the given Reporting Period or an information return that seeks to amend, correct or delete previously sent information. The possible values are:

  • DPI401 = The message contains new information
  • DPI402 = The message contains corrections/deletions of previously sent information
  • DPI403 = The message advises that there is no data to report (This value must not be used)

Corrections, amendments or deletions of previously sent data should never be included with new information returns. For more details, please see the Modifying previously filed returns page.

Size:
enum
Input type:
dpi:DPIMessageTypeIndic_EnumType
Requirement:
Mandatory

ReportingPeriod data element

Identifies the end of the calendar year to which the information in the DPIR message relates. It is to be provided in YYYY-MM-DD format. For example, if the information relates to the 2022 Reportable Period, the field would read ʻ2022-12-31’.

Input type:
xsd:dateTime
Requirement:
Mandatory

Timestamp data element

Identifies the date and time when the message was compiled. The format is YYYY-MM-DD’T’hh:mm:ss.nnn. Fractions of seconds may be used (in such a case, the milliseconds will be provided with three digits). Examples: ʻ2015-03-15T09:45:30ʼ or ʻ2015-03-15T09:45:30.789ʼ (with milliseconds)

Input type:
xsd:dateTime
Requirement:
Mandatory

Platform Operator data element

The Platform Operator data element is of the Correctable Platform Operator (CorrectablePlatformOperator) type, which extends the Organisation Party (OrganisationParty) type and adds the Document Specification data element to it. For more details about the Document Specification data element, please refer to the Identification of data elements section.

The Organisation Party type defines the structure that represents either a Platform Operator or an Entity Seller.

Image placeholder: Figure 3: Organization Party type

The following are descriptions of elements comprising the Organization Party type.

URL attribute

Specifies the URL of the Platform Operator or an Entity Seller. While this field is optional, Platform Operators are expected to have it, while not all Entity Sellers are expected to have it.

Size:
1 to 200 characters
Input type:
stf:StringMin1Max200_Type
Requirement:
Optional

ResCountryCode data element

Repeatable data element which specifies the tax residency country of an entity. An entity’s tax residency country is determined based on its Primary Address. Additionally, Residency Country Codes may be obtained using a tax administration’s API-based service, if and/or when it is available.

Size:
2-character
Input type:
iso:CountryCode_Type
Requirement:
Optional

TIN data element

Provides the identification number used by the CRA and corresponds to a BN9 (defined in the Definition of key terms section). In case the entity does not have a TIN, or the TIN is not known, the value ‘NOTIN’ should be entered and the unknown attribute (see below) must be set to ‘true’.

Size:
1 to 200 characters
Input type:
dpi:TIN_Type
Requirement:
Mandatory

issuedBy attribute

Specifies the jurisdiction that issued the TIN, unless the TIN element is flagged as ‘unknown’. It is used for future alignment with the Model Rules.

Size:
2-character
Input type:
iso:CountryCode_Type
Requirement:
Optional

Unknown attribute

Can be provided if the TIN is not available or non-existent. Any value provided for a TIN flagged as unknown will be discarded.

Size:
1-character
Input type:
xsd:Boolean
Requirement:
Optional

IN data element

Repeatable data element and may be provided if additional identifiers (other than the BN9 provided in the TIN or BN15 provided in the VAT data element) are available.

Size:
1 to 200 characters
Input type:
OrganisationIN_Type
Requirement:
Optional

OrganisationIN_Type is a complex data type which extends stf:StringMin1Max200_Type and adds two attributes: issuedBy and INType - description of these attributes follows below.

issuedBy attribute

Optional attribute and describes the jurisdiction that issued the IN.

Size:
2-character
Input type:
iso:CountryCode_Type
Requirement:
Optional

INType attribute

Type of identification number being sent. It can contain one of the following values:

  • IIN = represents an individual identification number
  • LEI = represents a legal entity identifier
  • EIN = represents an entity identification number
  • BRN = represents a business registration number
  • Other
Size:
enum
Input type:
dpi:INType_EnumType
Requirement:
Optional

VAT data element

GST/HST Registration Number. This corresponds to the BN15.

Size:
1 to 200 characters
Input type:
stf:StringMin1Max200_Type
Requirement:
Optional

Name data element

The legal name of the entity, as indicated in its articles of incorporation or any similar document

Size:
1 to 200 characters
Input type:
dpi:NameOrganisation_Type
Requirement:
Mandatory

Note: In case of Entity Sellers, the Name data element may be used for the legal names and operating names – both should be provided if available. If Name does not exist, then the value of ‘UNKNOWN’ must be entered.

In the case of Platform Operators, the Name data element is to be used for their legal name only.

PlatformBusinessName data element

Business or operating name(s) of the Platform(s) in respect of which the Platform Operator is reporting. This element must not be used for Entity Sellers.

Size:
1 to 200 characters
Input type:
stf:StringMin1Max200_Type
Requirement:
Optional

Address data element

Entity’s address. It should include at least the Primary Address. Additional addresses should be provided, if available.

Input type:
dpir:Address_Type
Requirement:
Required

DocSpec data element

Provides a mechanism to uniquely identify a Platform Operator or Reportable Seller data elements, not only within the current DPIR message but across all DPIR messages, all Platform Operators and all Reportable Periods. The Document Specification data element is detailed in the Identification of data elements section.

Input type:
sft:DocSpec_Type
Requirement:
Mandatory
